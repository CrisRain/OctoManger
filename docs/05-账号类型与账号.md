# 账号类型与账号

OctoManger 里的“账号”分两类：

- 普通业务账号：保存在 `accounts` 表，只支持 `generic` 类型
- 邮箱账号：保存在 `email_accounts` 表，单独管理，见 [09-邮箱账号](09-邮箱账号.md)

## 概念关系

关系可以理解为：

1. `Account Type` 定义一种账号类型
2. `Account` 是该类型下的具体账号实例
3. `Job` 和 `Trigger` 都引用 `type_key + action_key`
4. `OctoModule` 脚本按 `type_key` 解析入口文件

## Account Type

接口：

- `GET /api/v1/account-types/`
- `GET /api/v1/account-types/{key}`
- `POST /api/v1/account-types/`
- `PATCH /api/v1/account-types/{key}`
- `DELETE /api/v1/account-types/{key}`

核心字段：

| 字段 | 说明 |
| --- | --- |
| `key` | 类型唯一标识，例如 `github` |
| `name` | 展示名 |
| `category` | `system`、`email`、`generic` |
| `schema` | 账号 `spec` 的 JSON Schema 风格描述 |
| `capabilities` | 可执行动作、可选 daemon 能力 |
| `script_config` | 模块入口路径配置，仅对 `generic` 有意义 |
| `version` | 每次更新递增 |

### `category` 的区别

| category | 用途 |
| --- | --- |
| `generic` | 可创建普通账号、可执行 Job、可绑定 OctoModule |
| `email` | 预留类别，邮箱能力实际走专用资源 |
| `system` | 系统内部用途，不参与普通账号和模块执行 |

## 创建 `generic` 类型时会发生什么

当你创建或更新一个 `generic` 类型时，后端会：

1. 校验 `schema`、`capabilities`、`script_config`
2. 解析模块入口路径
3. 自动确保入口脚手架文件存在

默认入口路径：

```text
scripts/python/modules/<type_key>/main.py
```

所以新建一个 `generic` 类型后，模块目录通常会自动生成。

## `schema` 的定位

`schema` 主要用于：

- 界面渲染和录入约束表达
- 约定账号 `spec` 中有哪些字段

后端当前只强校验：

- `schema` 必须是一个合法 JSON 对象
- `spec` 本身也必须是合法 JSON 对象

也就是说，真正的业务校验通常仍然由模块脚本自己完成。

## `capabilities` 的定位

最常见的写法：

```json
{
  "actions": [
    { "key": "VERIFY" },
    { "key": "REGISTER" }
  ]
}
```

如果要启用 daemon：

```json
{
  "actions": [
    { "key": "VERIFY" }
  ],
  "daemon": {
    "action": "WATCH"
  }
}
```

说明：

- `actions` 用于控制台和调用方感知可执行动作
- daemon 进程只会接管 `status = 1` 的账号

## `script_config`

`script_config` 用来覆盖默认入口文件。

支持的入口键名包括：

- `entry`
- `module_entry`
- `module_script`
- `script`
- `path`
- `file`

也支持嵌套在 `octoModule` 下。

示例：

```json
{
  "entry": "github/main.py"
}
```

限制：

- 必须是相对路径
- 必须位于 `paths.octo_module_dir` 内
- 不能用路径穿越跳出模块目录

## Account

接口：

- `GET /api/v1/accounts/`
- `POST /api/v1/accounts/`
- `GET /api/v1/accounts/{id}`
- `PATCH /api/v1/accounts/{id}`
- `DELETE /api/v1/accounts/{id}`
- `POST /api/v1/accounts/batch-patch`
- `POST /api/v1/accounts/batch-delete`

核心字段：

| 字段 | 说明 |
| --- | --- |
| `type_key` | 关联的 `generic` 类型 |
| `identifier` | 类型内唯一标识 |
| `status` | 小整数状态位 |
| `tags` | 文本标签数组 |
| `spec` | 账号业务数据 |

### 状态语义

当前界面通常把：

- `0` 显示为未启用 / 待验证
- `1` 显示为已启用 / 已验证

但要注意：

- 普通 Job 不会自动按 `status` 过滤账号
- daemon 进程会只加载 `status = 1` 的账号

所以 `status` 更像业务状态位，不是全局执行开关。

### 唯一性

同一个 `type_key` 下，`identifier` 必须唯一。

## 批量操作

支持的批量能力：

- 批量改状态
- 批量改标签
- 批量删除

这些操作不是同步逐条返回，而是：

1. 创建一条系统批量 Job
2. 异步入队给 Worker
3. 最终结果写入 `job_runs`

## 最小示例

### 创建类型

```json
{
  "key": "demo",
  "name": "Demo",
  "category": "generic",
  "schema": {
    "type": "object",
    "properties": {
      "token": { "type": "string" }
    }
  },
  "capabilities": {
    "actions": [
      { "key": "VERIFY" }
    ]
  },
  "script_config": {}
}
```

### 创建账号

```json
{
  "type_key": "demo",
  "identifier": "alice",
  "status": 1,
  "tags": ["seed"],
  "spec": {
    "token": "demo-token"
  }
}
```

## 删除限制

删除 Account Type 时，如果仍有账号或其他资源引用它，数据库层可能返回冲突，接口会转成“仍在使用中”的错误。

在删类型前，先确认：

- 该类型下的账号是否已清理
- 相关 Job / Trigger 是否不再使用这个 `type_key`
