# 快速开始

这份文档用于把项目从空环境启动到“成功跑通一条 OctoModule 任务”。

## 前置要求

推荐环境：

- Docker Desktop 或 Docker Engine
- Go 1.22+
- Bun
- Python 3.10+

运行时依赖：

- PostgreSQL
- Redis

## 方式一：Docker Compose

最省事的启动方式：

```bash
docker compose up -d --build
```

启动完成后访问：

- `https://localhost:8080`
- 或 `http://localhost:8080`，会自动跳到 HTTPS

注意：

- 首次访问会遇到自签名证书提示，允许继续即可。
- Compose 默认把前端构建产物和后端 API 统一托管在一个进程里。

## 方式二：本地开发

先启动数据库和 Redis：

```bash
docker compose up -d postgres redis
```

再启动后端：

```bash
cd backend
SERVER_TLS=false go run ./cmd/octomanger
```

如果你只想调试部分服务，也可以这样启动：

```bash
cd backend
SERVER_TLS=false go run ./cmd/octomanger -services=api,worker
```

前端开发服务器：

```bash
cd web
bun dev
```

访问地址：

- 前端开发页：`http://localhost:5173`
- 后端 API：`http://localhost:8080`

本地调试时建议把 `SERVER_TLS=false` 打开，因为 Vite 代理默认指向 `http://localhost:8080`。

## 第一步：初始化系统

首次启动后，系统会处于 `needs_setup=true` 状态。

可以通过控制台初始化，也可以直接调用接口：

```bash
curl -X POST http://localhost:8080/api/v1/system/setup \
  -H "Content-Type: application/json" \
  -d "{\"admin_key_name\":\"Admin Key\"}"
```

返回值里会带上：

- `api_key`：仅包含前缀和元信息
- `raw_key`：真正的 Admin Key，只会返回这一次

把 `raw_key` 妥善保存，后续所有管理接口都要通过 `X-Api-Key` 传递。

## 第二步：创建一个最小模块

建议先创建一个 `generic` 类型，系统会自动生成脚手架文件。

示例 Account Type：

```json
{
  "key": "demo",
  "name": "Demo",
  "category": "generic",
  "schema": {
    "type": "object",
    "properties": {
      "token": { "type": "string", "title": "Token" }
    }
  },
  "capabilities": {
    "actions": [
      { "key": "VERIFY" }
    ]
  },
  "script_config": {}
}
```

创建后，系统会确保默认入口存在：

```text
scripts/python/modules/demo/main.py
```

## 第三步：创建一个账号

示例账号：

```json
{
  "type_key": "demo",
  "identifier": "alice",
  "status": 1,
  "tags": ["seed"],
  "spec": {
    "token": "demo-token"
  }
}
```

`identifier` 在同一个 `type_key` 下必须唯一。

## 第四步：先做一次 Dry Run

推荐先在控制台 `Modules -> demo -> Dry Run` 验证脚本能否正常执行。

等价接口：

```json
POST /api/v1/octo-modules/demo:dry-run
{
  "action": "VERIFY",
  "account": {
    "identifier": "alice",
    "spec": {
      "token": "demo-token"
    }
  },
  "params": {}
}
```

Dry Run 成功后，再创建正式 Job。

## 第五步：创建一条正式任务

示例：

```json
{
  "type_key": "demo",
  "action_key": "VERIFY",
  "selector": {
    "identifiers": ["alice"]
  },
  "params": {}
}
```

创建成功后：

- Job 状态初始为 `0`（queued）
- Worker 会异步拉起执行
- 结果写入 `job_runs`

## 第六步：查看结果

控制台里可以直接看 `Jobs` 和 `Modules -> 运行历史`。

接口查询：

```bash
curl -H "X-Api-Key: <admin-key>" http://localhost:8080/api/v1/jobs/
curl -H "X-Api-Key: <admin-key>" http://localhost:8080/api/v1/jobs/1
curl -H "X-Api-Key: <admin-key>" http://localhost:8080/api/v1/octo-modules/demo/runs
```

## 推荐上手顺序

建议按这个顺序熟悉系统：

1. 跑通 `demo` 类型的 `VERIFY` Dry Run
2. 跑通正式 Job
3. 理解 `selector`、`params` 和 `JobRun`
4. 再接入 Trigger
5. 最后接入邮箱和更复杂的 OctoModule

## 常见问题

### 打开不了页面

优先检查：

- `docker compose ps`
- `curl http://localhost:8080/healthz`
- 本地开发时是否把 `SERVER_TLS=false` 打开

### 创建了 generic 类型但找不到脚本

默认脚本路径是：

```text
scripts/python/modules/<type_key>/main.py
```

如果设置了 `script_config`，则以 `script_config` 里声明的相对路径为准。

### 前端能打开，但 API 请求失败

大多是这两类原因：

- 没有完成初始化，系统还处于 `needs_setup`
- 本地前端走的是 HTTP 代理，而后端却以 HTTPS 启动
