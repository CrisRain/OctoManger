# 架构说明

OctoManger 的核心目标不是“只管账号表”，而是把账号、模块、异步执行和外部触发统一成一个可运营的平台。

## 核心组件

### 1. React 控制台 `web/`

负责：

- 首次初始化
- Admin Key 登录
- 账号类型、账号、邮箱、Job、Trigger、模块、证书的管理界面
- 通过 REST API 调用后端

### 2. Go API `backend/cmd/api`

负责：

- 提供 REST API
- 托管前端静态文件
- 持久化数据库记录
- 管理系统配置、API Key、SSL 证书
- 提供 OctoModule 内部 API

### 3. Worker `backend/cmd/worker`

负责：

- 从 Redis / Asynq 拉取任务
- 执行 Job 和批量任务
- 写入 `job_runs`
- 调用 Python 模块

### 4. Scheduler `backend/cmd/scheduler`

负责：

- 轮询数据库中的 Job
- 找到 `params._schedule` 不为空的 Job
- 按 cron 表达式重复入队

### 5. Daemon `backend/cmd/daemon`

负责：

- 托管 daemon 能力的长驻 Python 进程
- 为每个 `status = 1` 的账号单独维持一个子进程
- 把 daemon 输出的事件写成 `job_run`

### 6. OctoModule `scripts/python/modules/`

负责：

- 承载业务动作实现
- 接收 JSON 输入
- 输出 success / error / log / session
- 通过 `octo.py` 回调内部 API

## 统一入口

`backend/cmd/octomanger` 可以一次拉起多种服务。

可选服务：

- `api`
- `worker`
- `scheduler`
- `daemon`

示例：

```bash
go run ./cmd/octomanger -services=api,worker
```

## Job 执行数据流

标准 Job 的路径如下：

1. 控制台或 API 创建一条 Job
2. API 把 Job 写入 PostgreSQL，状态置为 `queued`
3. API 通过 Asynq 把 `job:dispatch` 任务写入 Redis
4. Worker 取出任务后把 Job 状态更新为 `running`
5. Worker 按 `type_key` 找到对应 Account Type
6. Worker 解析模块入口路径
7. Worker 根据 `selector` 过滤账号
8. 对每个命中的账号执行 Python 模块
9. 每次执行生成一条 `job_run`
10. 全部成功则 Job 置为 `done`，中途失败则置为 `failed`

关键点：

- Job 只支持 `generic` 类型。
- 当前执行策略是串行遍历命中的账号。
- 某个账号失败后，整条 Job 会直接失败并停止后续账号。

## Trigger 数据流

Trigger 是“外部 Webhook -> 内部 Job”的映射层。

路径如下：

1. 外部系统请求 `/webhooks/{slug}`
2. 通过 Bearer Token 或 Webhook API Key 做鉴权
3. 后端读取 Trigger 配置
4. 把 `default_selector` 和请求里的 `selector` 合并
5. 把 `default_params` 和请求里的 `extra_params` 合并
6. 自动向参数里注入 `_trigger` 元数据
7. 根据模式：
   - `async`：创建 Job 并入队
   - `sync`：在 API 进程里直接执行并返回结果摘要

## 模块执行协议

Worker 向模块传入的请求本质上是一个 JSON 对象：

```json
{
  "action": "VERIFY",
  "account": {
    "identifier": "alice",
    "spec": {}
  },
  "params": {},
  "context": {
    "request_id": "1:2",
    "protocol": "ndjson.v1",
    "api_url": "http://127.0.0.1:8080",
    "api_token": "..."
  }
}
```

模块标准输出支持三类消息：

- `{"status":"log",...}`：实时日志
- `{"status":"success",...}`：成功结果
- `{"status":"error",...}`：失败结果

如果模块返回 `session`，Worker 会把它落到 `account_sessions`。

## 模块入口解析

`generic` 类型默认脚本路径：

```text
scripts/python/modules/<type_key>/main.py
```

也可以通过 `script_config` 指定相对路径。后端会校验：

- 必须是相对路径
- 必须位于 `paths.octo_module_dir` 内
- 不能使用路径穿越逃出模块目录

## HTTP、静态资源与 TLS

API 路由和前端静态资源由同一个 HTTP 服务器提供：

- `/api/...`：业务 API
- `/webhooks/...`：Trigger 入口
- `/healthz`：健康检查
- 其他 GET / HEAD 请求：优先尝试返回前端静态资源，否则回退到 `index.html`

TLS 行为：

- `server.tls=true` 时默认启用 HTTPS
- 如果数据库里没有证书，会自动生成自签名证书
- 明文 HTTP 请求会被同端口重定向到 HTTPS

## 核心数据对象

| 对象 | 作用 |
| --- | --- |
| `account_types` | 定义账号类型、字段 schema、模块能力 |
| `accounts` | `generic` 类型的业务账号 |
| `email_accounts` | 邮箱账号，独立于普通账号表 |
| `jobs` | 任务定义和状态 |
| `job_runs` | 单次执行结果、日志、错误信息 |
| `api_keys` | Admin、Webhook、Internal Key |
| `trigger_endpoints` | Webhook Trigger 配置 |
| `system_config` | 系统配置与证书内容 |
| `account_sessions` | 模块返回的 cookie / token 会话 |

## API 响应约定

统一响应结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

返回规则：

- `code = 0` 表示成功
- 业务错误通常仍然返回 HTTP 200，但 `code != 0`
- 明显的请求错误也可能直接返回 HTTP 4xx / 5xx

常见业务错误码：

- `40000`：参数错误
- `40100`：未授权
- `40400`：资源不存在
- `40900`：冲突
- `50000`：内部错误
