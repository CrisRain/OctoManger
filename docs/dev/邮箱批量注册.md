# 邮箱批量注册内部说明

## 当前状态

当前默认实现已经不是“只生成邮箱记录”。

`Outlook` 批量注册现在会：

1. 通过 Playwright 打开微软真实注册页
2. 逐个创建 `outlook.com / hotmail.com / live.com / msn.com` 账号
3. 在人机验证阶段等待人工在浏览器里完成验证
4. 注册成功后继续完成 Outlook OAuth 授权
5. 把可直接使用的 `graph_config` 落到 `email_accounts`

所以它现在是“真实外部邮箱注册 + OAuth 回收 + 入库”。

## 只支持 Outlook

当前仅支持这些域名：

- `outlook.com`
- `hotmail.com`
- `live.com`
- `msn.com`

其他 provider 目前会直接报错，不再回退到旧的假生成逻辑。

## 调用链

入口接口：

```text
POST /api/v1/email/accounts/batch-register
```

主链路：

1. `service.EmailAccountService.BatchRegister`
2. `createAndEnqueueAsyncJob(...)`
3. `task.Producer.EnqueueBatchEmailRegister`
4. `taskhandler.BatchHandler.ProcessEmailBatchRegister`
5. `service.OutlookEmailBatchRegistrar.Prepare`
6. `scripts/python/modules/_email_batch_outlook/main.py`
7. Worker 将返回的真实 `graph_config` 写入 `email_accounts`

## 前端提交的关键参数

顶层仍使用 `dto.BatchRegisterEmailRequest`，但真实注册的核心配置都放在 `options`：

| 字段 | 说明 |
| --- | --- |
| `password` | 批量注册时使用的统一密码，必填 |
| `first_name` / `last_name` | 注册页姓名 |
| `birth_month` / `birth_day` / `birth_year` | 注册页生日 |
| `headless` | 是否隐藏浏览器；默认建议关闭，即显示浏览器窗口 |
| `challenge_timeout_seconds` | 人机验证等待超时 |
| `oauth.client_id` | Outlook OAuth Client ID |
| `oauth.client_secret` | 可选 |
| `oauth.tenant` | 一般为 `consumers` |
| `oauth.redirect_uri` | OAuth 回调地址 |
| `oauth.scope` | 必须包含 `offline_access`，否则拿不到 refresh token |
| `oauth.mailbox` | 默认邮箱目录 |
| `oauth.graph_base_url` | Graph API Base URL |

## 现实限制

这套实现仍然受微软注册页风控影响，尤其是：

- 人机验证
- 手机验证
- IP / 代理信誉
- OAuth 应用配置是否正确

所以它不是“完全无人工”的批量注册器，而是“真实注册流程的自动化执行器”，默认设计就是在人机验证时允许人工介入。
