# 部署与配置

本项目支持两种常见运行方式：

- Docker Compose：最适合直接体验和单机部署
- 本地开发：最适合调试前端和后端代码

## Docker Compose

启动：

```bash
docker compose up -d --build
```

查看状态：

```bash
docker compose ps
docker compose logs -f app
```

默认暴露：

- PostgreSQL：`5432`
- Redis：`6379`
- 应用：`8080`

访问方式：

- 推荐 `https://localhost:8080`
- `http://localhost:8080` 会自动重定向到 HTTPS

## 数据卷

Compose 默认使用以下持久化卷：

| 卷 | 用途 |
| --- | --- |
| `pgdata` | PostgreSQL 数据 |
| `redisdata` | Redis AOF 数据 |
| `pip_cache` | Python 依赖缓存 |
| `playwright_cache` | Playwright 浏览器缓存 |

停止与清理：

```bash
docker compose down
docker compose down -v
```

`scripts/python/modules/` 默认通过 bind mount 挂进容器，方便在宿主机和容器之间共享同一份模块脚本。

`-v` 会同时清空数据库、Redis、pip 和 Playwright 浏览器缓存。

## Docker + Playwright

如果主应用跑在 Docker 里，而某些 OctoModule 依赖 Playwright、人工验证码或宿主机浏览器环境，推荐把 OctoModule 服务改成宿主机独立进程：

### 1. 让容器改连外部 octomodule service

可以在 `.env` 或 shell 里设置：

```bash
OCTOMODULE_SERVICE_URL=http://host.docker.internal:8091
OCTOMODULE_SERVICE_EMBEDDED=false
```

然后启动：

```bash
docker compose up -d --build
```

其中：

- `OCTOMODULE_SERVICE_EMBEDDED=false` 表示容器内不再启动内嵌 octomodule service
- `OCTOMODULE_SERVICE_URL` 指向宿主机上的独立 octomodule service

Compose 已经内置 `host.docker.internal` 映射，并且 `scripts/python/modules/` 会直接挂载到容器中，方便宿主机和容器共享模块代码。

如果你必须在容器内安装 Chromium，而不是改成宿主机 octomodule service，建议额外配置 Playwright 下载镜像或代理：

```env
PLAYWRIGHT_DOWNLOAD_HOST=https://<your-playwright-mirror>
PLAYWRIGHT_DOWNLOAD_CONNECTION_TIMEOUT=120000
PLAYWRIGHT_BROWSERS_PATH=/ms-playwright
# 如需代理：
# HTTP_PROXY=http://host.docker.internal:7890
# HTTPS_PROXY=http://host.docker.internal:7890
```

`docker-compose.yml` 已经透传了这些环境变量，并把浏览器缓存目录持久化到 `playwright_cache`，避免每次重新下载。

### 2. 在宿主机启动 octomodule service

```powershell
cd backend
$env:PATHS_OCTO_MODULE_DIR="..\scripts\python\modules"
$env:PYTHON_BIN="python"
go run ./cmd/octomodule
```

这样：

- API / Worker / Scheduler / Daemon 继续跑在 Docker 里
- OctoModule 的 Playwright 执行改在宿主机环境
- 容器内任务仍然通过 `octomodule_service.url` 远程调用模块

## 本地开发

### 1. 先启动依赖

```bash
docker compose up -d postgres redis
```

### 2. 启动后端

统一入口：

```bash
cd backend
SERVER_TLS=false go run ./cmd/octomanger
```

只启动部分服务：

```bash
cd backend
SERVER_TLS=false go run ./cmd/octomanger -services=api,worker
```

独立进程方式：

```bash
cd backend
SERVER_TLS=false go run ./cmd/api
go run ./cmd/worker
go run ./cmd/scheduler
go run ./cmd/daemon
```

### 3. 启动前端

```bash
cd web
bun dev
```

Vite 开发服务器默认：

- 前端地址：`http://localhost:5173`
- 代理 `/api/v1` 和 `/healthz` 到 `http://localhost:8080`

因此本地前后端联调时，推荐让后端走明文 HTTP，也就是设置 `SERVER_TLS=false`。

## 构建与测试

后端测试：

```bash
go test ./backend/...
```

前端测试：

```bash
cd web
bun run test:run
```

前端构建：

```bash
cd web
bun run build
```

## 配置加载顺序

配置来源按优先级从低到高理解即可：

1. 程序内默认值
2. `config.yaml`
3. 环境变量

指定自定义配置文件：

```bash
CONFIG_FILE=/abs/path/custom.yaml go run ./cmd/octomanger
```

默认会尝试从这些位置找 `config.yaml`：

- 当前目录
- `configs/`
- 上级目录
- 上级目录的 `configs/`

## 关键配置项

| 配置键 | 环境变量 | 说明 |
| --- | --- | --- |
| `server.port` | `SERVER_PORT` | API 监听端口 |
| `server.tls` | `SERVER_TLS` | 是否启用 TLS，默认 `true` |
| `server.mode` | `SERVER_MODE` | Gin 运行模式 |
| `server.web_dist_dir` | `SERVER_WEB_DIST_DIR` | 前端构建产物目录 |
| `database.url` | `DATABASE_URL` | PostgreSQL 连接串 |
| `database.auto_migrate` | `DATABASE_AUTO_MIGRATE` | 启动时自动迁移 |
| `database.reset` | `DATABASE_RESET` | 迁移时是否删除旧结构 |
| `redis.addr` | `REDIS_ADDR` | Redis 地址 |
| `asynq.redis_addr` | `ASYNQ_REDIS_ADDR` | Asynq 单独 Redis 地址，不填时沿用 `redis.addr` |
| `asynq.concurrency` | `ASYNQ_CONCURRENCY` | Worker 并发数 |
| `python.bin` | `PYTHON_BIN` | Python 可执行文件 |
| `python.timeout_seconds` | `PYTHON_TIMEOUT_SECONDS` | 模块执行超时时间 |
| `octomodule_service.url` | `OCTOMODULE_SERVICE_URL` | Worker / API 调用的 OctoModule 服务地址 |
| `octomodule_service.embedded` | `OCTOMODULE_SERVICE_EMBEDDED` | 统一入口是否在本进程内启动 OctoModule 服务，默认 `true` |
| `octomodule_service.token` | `OCTOMODULE_SERVICE_TOKEN` | 远程 OctoModule 服务鉴权 token |
| `paths.octo_module_dir` | `PATHS_OCTO_MODULE_DIR` | OctoModule 根目录 |
| `logging.file` | `LOGGING_FILE` | 日志文件路径 |
| `logging.level` | `LOGGING_LEVEL` | 日志级别 |
| `logging.format` | `LOGGING_FORMAT` | `console` 或 `json` |

## 关于 `python.script`

配置文件里仍然保留了 `python.script` 字段，但当前 `generic` 模块执行流程实际使用的是每个类型解析出来的模块入口文件，例如：

```text
scripts/python/modules/github/main.py
```

因此：

- 常规 Job / Dry Run / Trigger 不依赖 `python.script`
- 重点配置应放在 `paths.octo_module_dir`

## TLS 与证书

当 `server.tls=true` 时：

- 系统默认启用 HTTPS
- 若数据库里没有证书，会自动生成自签名证书
- HTTP 请求会被同端口重定向到 HTTPS

上传正式证书可以通过：

- 控制台 `SSL` 页面
- `PUT /api/v1/ssl/certificate`

更详细说明见 [10-系统设置与SSL](10-系统设置与SSL.md)。

## 典型环境变量示例

### Docker / 单机

```bash
SERVER_PORT=8080
SERVER_TLS=true
SERVER_WEB_DIST_DIR=/app/web-dist
DATABASE_URL=postgres://octo:octo@postgres:5432/octomanger?sslmode=disable
REDIS_ADDR=redis:6379
PATHS_OCTO_MODULE_DIR=/app/scripts/python/modules
PYTHON_BIN=python3
```

### Docker 主应用 + 宿主机 OctoModule 服务

```bash
SERVER_PORT=8080
SERVER_TLS=true
DATABASE_URL=postgres://octo:octo@postgres:5432/octomanger?sslmode=disable
REDIS_ADDR=redis:6379
PATHS_OCTO_MODULE_DIR=/app/scripts/python/modules
OCTOMODULE_SERVICE_URL=http://host.docker.internal:8091
OCTOMODULE_SERVICE_EMBEDDED=false
```

```bash
SERVER_PORT=8080
SERVER_TLS=false
DATABASE_URL=postgres://octo:octo@127.0.0.1:5432/octomanger?sslmode=disable
REDIS_ADDR=127.0.0.1:6379
PATHS_OCTO_MODULE_DIR=../scripts/python/modules
PYTHON_BIN=python
```
