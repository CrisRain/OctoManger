# 任务系统

OctoManger 的任务系统由三部分组成：

- `jobs`：任务定义和状态
- `worker`：异步执行器
- `job_runs`：执行明细

## Job 是什么

一个 Job 表示“对某一类账号执行某个动作”。

核心字段：

| 字段 | 说明 |
| --- | --- |
| `type_key` | 目标账号类型，必须是 `generic` |
| `action_key` | 要执行的动作 |
| `selector` | 账号筛选条件 |
| `params` | 传给模块的动作参数 |
| `status` | 当前状态 |

创建接口：

```json
POST /api/v1/jobs/
{
  "type_key": "github",
  "action_key": "VERIFY",
  "selector": {
    "identifiers": ["alice"]
  },
  "params": {}
}
```

## Job 状态

状态枚举：

| 值 | 含义 |
| --- | --- |
| `0` | queued |
| `1` | running |
| `2` | done |
| `3` | failed |
| `4` | canceled |

注意：

- 新建 Job 后默认是 `0`
- Worker 开始处理后会改成 `1`
- 任何一个账号执行失败，整条 Job 会变成 `3`
- 用户取消后会变成 `4`

## selector 语法

当前支持的筛选键：

| selector 字段 | 说明 |
| --- | --- |
| `account_ids` | 指定账号 ID 列表 |
| `identifiers` | 指定账号 identifier 列表 |
| `identifier_contains` | identifier 包含某个子串 |
| `limit` | 取前 N 个命中账号 |

示例：

```json
{
  "identifier_contains": "seed-",
  "limit": 20
}
```

说明：

- selector 只基于账号集合做过滤
- 当前不会按 `status` 自动过滤账号

## Worker 执行规则

执行顺序：

1. 读取 `type_key` 对应的全部账号
2. 用 selector 过滤
3. 逐个执行模块
4. 为每个账号写一条 `job_run`

失败策略：

- 如果一个账号失败，当前 Job 直接失败
- 后续账号不再继续执行

空集策略：

- 如果 selector 命中 0 个账号，Job 会直接完成为 `done`

## `job_runs`

`job_runs` 是执行明细表。

每条记录通常包含：

- `job_id`
- `account_id`
- `worker_id`
- `result`
- `logs`
- `error_code`
- `error_message`
- `started_at`
- `ended_at`

你会在这些场景里看到 `job_run`：

- 普通 Job 执行
- Trigger 的同步执行摘要
- 批量操作完成摘要
- daemon 事件落库

## 常用接口

查询列表：

- `GET /api/v1/jobs/`
- `GET /api/v1/jobs/{id}`

操作：

- `POST /api/v1/jobs/{id}:cancel`
- `POST /api/v1/jobs/{id}:retry`
- `DELETE /api/v1/jobs/{id}`

重试语义：

- `retry` 会复制原 Job 的 `type_key`、`action_key`、`selector`、`params`
- 然后创建一条新的 Job 并重新入队

## 批量任务

以下批量操作不会同步逐条返回，而是创建系统批量 Job：

- 账号批量改状态 / 改标签 / 删除
- 邮箱批量删除 / 验证 / Graph 导入 / 批量生成记录

这些批量任务的结果通常汇总在单条 `job_run.result` 里，例如：

```json
{
  "total": 10,
  "success": 8,
  "failed": 2,
  "canceled": false
}
```

## 调度任务

Scheduler 不单独管理另一张“计划任务表”，而是直接从 Job 的 `params` 里读取 `_schedule`。

示例：

```json
{
  "_schedule": "0 */6 * * *",
  "force": true
}
```

含义：

- 只要 Scheduler 进程在运行
- 它就会每分钟扫描一次数据库
- 找到带 `_schedule` 的 Job 并按 cron 表达式重复入队

这意味着原 Job 同时兼具：

- 一次性的任务定义
- 周期调度模板

## 取消语义

取消接口只是把 Job 状态改成 `canceled`。

Worker 的行为是：

- 在开始执行前检查一次
- 每处理完一个账号前再检查一次

所以取消不是强杀当前 Python 进程，而是“尽快在安全点停止后续处理”。

## 超时与重试

Asynq 层默认设置：

- `job:dispatch`：最多重试 3 次，超时 5 分钟
- 批量任务：最多重试 1 次，超时 10 分钟

模块执行层还有一层 Python 超时：

- 由 `python.timeout_seconds` 控制
- 默认 60 秒

## 建议

- 开发模块时先做 Dry Run，不要直接拿大批量账号试错
- 需要周期调度时，把 `_schedule` 写进 `params`，并确保 Scheduler 进程已启动
- 把可观测信息写进模块日志，而不是只依赖最终结果
- 对长批量任务做好取消和幂等设计
